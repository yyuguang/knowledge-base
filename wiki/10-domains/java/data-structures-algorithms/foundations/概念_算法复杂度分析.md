# 概念：算法复杂度分析

## 定义

复杂度分析描述资源消耗随输入规模 `n` 增长的趋势。通常讨论时间复杂度与辅助空间复杂度，并用 `O`（渐近上界）、`Ω`（渐近下界）和 `Θ`（渐近紧界）表达增长级别。日常方案比较中，`O` 记号最常用，但不等于某次运行的精确耗时。

## 为什么重要

它让“这个实现快不快”变成可比较的问题：在相同机器和输入模型下，`O(n log n)` 通常会比 `O(n²)` 更能承受大规模数据；同时也能暴露以额外内存换取时间的取舍。

## 使用场景

- 为数组、链表、哈希表和树的操作选择合适实现。
- 判断排序、搜索、遍历等算法在目标数据量下是否可用。
- 审查 Java 代码中嵌套循环、重复扫描、递归调用和临时集合带来的成本。

## 核心机制

1. 定义输入规模，例如数组长度、节点数、边数或字符串长度。
2. 找出主导操作及其执行次数，忽略常数因子和低阶项。
3. 分别说明最好、平均、最坏情形；若依赖输入分布或哈希质量，应写明假设。
4. 单独统计辅助空间；递归还要计入调用栈深度。

常见增长级别：`O(1)`、`O(log n)`、`O(n)`、`O(n log n)`、`O(n²)`、`O(2ⁿ)`、`O(n!)`。其中后两类通常只适用于较小规模或必须通过剪枝/状态压缩的场景。

## 例子

在长度为 `n` 的数组中，顺序查找最坏需检查 `n` 个元素，故为 `O(n)`；若数组有序且可随机访问，二分查找每次将候选区间缩小一半，故为 `O(log n)`。但把二分查找直接用于链表时，定位中点本身并非 `O(1)`，不能只套用结论。

## 常见误区

- 只看循环层数，不看内层循环实际迭代次数或递归分支。
- 把 `O(1)` 误解为没有成本；哈希操作的平均 `O(1)` 仍依赖装载因子、散列分布和冲突处理。
- 只统计运行时间，忽略复制数组、缓存状态、递归栈等空间成本。
- 用复杂度替代基准测试；小规模数据、JVM 预热、对象分配和缓存局部性仍会影响实际性能。

## 我的理解

复杂度分析的价值不在于背诵符号，而在于先明确“操作是什么、输入规模是什么、哪些假设成立”，再做结构和算法选择。代码审查时，先问是否存在不必要的全量扫描，往往比立即微调语法更有效。
