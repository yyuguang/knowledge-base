# 概念_类加载机制

## 定义

Java 虚拟机把描述类的数据从 Class 文件加载到内存，并对数据进行校验、转换解析和初始化，最终形成可被虚拟机直接使用的 Java 类型。与 C/C++ 不同，Java 的类型加载、连接和初始化均在运行期完成，这为 Java 提供了极高的扩展性和灵活性。

## 生命周期 7 阶段

![[Attachments/jvm-classloading-lifecycle.excalidraw]]

加载（Loading）→ 验证（Verification）→ 准备（Preparation）→ 解析（Resolution）→ 初始化（Initialization）→ 使用（Using）→ 卸载（Unloading）

### 前 5 阶段详解
1. **加载**：通过全限定名获取二进制字节流 → 转为方法区数据结构 → 堆中生成 Class 对象
2. **验证**：文件格式 → 元数据 → 字节码（StackMapTable）→ 符号引用
3. **准备**：为 static 变量分配内存并设零值（final static 直接赋初始值）
4. **解析**：符号引用 → 直接引用（类/接口/字段/方法/方法句柄/方法类型/调用点限定符）
5. **初始化**：执行 `<clinit>`，合并所有 static 赋值 + static 块，JVM 保证线程安全

#### `<clinit>` 线程安全与单例模式

`<clinit>` 由 JVM 在 Class 对象上加锁保证线程安全，这是**静态内部类单例**的底层基础——多个线程并发调用 `getInstance()` 时，只有一个线程执行 `<clinit>` 完成 `InstanceHolder` 类的加载。

```java
/**
 * 静态内部类单例 —— 利用 <clinit> 的线程安全保证
 * JVM 在初始化 InstanceHolder 时自动加锁，无需 synchronized
 */
public class Singleton {
    private Singleton() {}

    private static class InstanceHolder {
        // <clinit> 合并 static 赋值，JVM 保证原子性
        static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return InstanceHolder.INSTANCE;  // 首次触发 InstanceHolder 的 <clinit>
    }
}
```

验证方法：在 `Singleton()` 构造器中加 `Thread.sleep(2000)`，用 20 个线程并发调用 `getInstance()`，所有线程拿到的 INSTANCE 引用完全相同，无一新建。

## 6 种主动初始化触发

- `new` / `getstatic` / `putstatic` / `invokestatic` 指令
- `java.lang.reflect` 反射
- 子类初始化触发父类
- JVM 启动时指定 main 类
- `java.lang.invoke.MethodHandle` 解析 REF_getStatic 等
- 默认方法（JDK 8+）

## 类加载器与双亲委派模型

- **Bootstrap ClassLoader** → **Extension ClassLoader** → **Application ClassLoader** → **自定义 ClassLoader**
- **优点**：基础类统一（如 java.lang.Object 保证一致），避免重复加载
- **破坏场景**：JDK 1.2 兼容、SPI（线程上下文类加载器）、OSGi（网状结构）
- **JDK 9 模块化（Jigsaw）**：类加载器结构变化，平台类加载器取代 Extension

### 打印 ClassLoader 层级

```java
/**
 * 打印当前类、核心类、SPI 接口的类加载器层级
 * 输出示例：
 *   AppClassLoader -> PlatformClassLoader -> null (Bootstrap)
 */
public class ClassLoaderHierarchy {
    public static void main(String[] args) {
        // 自定义类
        printLoader(ClassLoaderHierarchy.class);
        // 核心类：Bootstrap ClassLoader 加载，返回 null
        printLoader(String.class);
        // SPI 接口类（rt.jar 在 JDK8，需线程上下文加载器）
        printLoader(java.sql.Driver.class);
    }

    static void printLoader(Class<?> clz) {
        ClassLoader loader = clz.getClassLoader();
        System.out.print(clz.getSimpleName() + " <- ");
        while (loader != null) {
            System.out.print(loader.getClass().getSimpleName() + " <- ");
            loader = loader.getParent();
        }
        System.out.println("Bootstrap (null)");
    }
}
```

### 自定义 ClassLoader：打破双亲委派，从文件系统加载

```java
import java.io.*;

/**
 * 打破双亲委派的 ClassLoader：
 * 重写 loadClass 而非 findClass，跳过父加载器的委托链
 * 从指定文件系统路径读取 .class 文件
 *
 * 注意：生产环境慎用，会破坏 Java 核心类的隔离保证
 */
public class PathClassLoader extends ClassLoader {
    private final String classPath;

    public PathClassLoader(String classPath) {
        super(null);  // 父加载器置 null，彻底打破双亲委派
        this.classPath = classPath;
    }

    @Override
    public Class<?> loadClass(String name) throws ClassNotFoundException {
        // 只让自己加载目标类，其余仍委托父加载器
        if (name.startsWith("com.yourapp")) {
            return findClass(name);
        }
        return super.loadClass(name);  // java.lang.* 等核心类仍走双亲委派
    }

    @Override
    protected Class<?> findClass(String name) throws ClassNotFoundException {
        byte[] classBytes = loadClassBytes(name);
        return defineClass(name, classBytes, 0, classBytes.length);
    }

    private byte[] loadClassBytes(String fullName) throws ClassNotFoundException {
        String fileName = classPath + File.separator
            + fullName.replace('.', File.separatorChar) + ".class";
        try (ByteArrayOutputStream baos = new ByteArrayOutputStream();
             FileInputStream fis = new FileInputStream(fileName)) {
            byte[] buffer = new byte[4096];
            int bytesRead;
            while ((bytesRead = fis.read(buffer)) != -1) {
                baos.write(buffer, 0, bytesRead);
            }
            return baos.toByteArray();
        } catch (IOException e) {
            throw new ClassNotFoundException("Fail to load: " + fileName, e);
        }
    }
}
```

### 线程上下文类加载器：SPI 模式

SPI（Service Provider Interface）中，接口定义在 `rt.jar`（Bootstrap 加载），实现类在 `classpath`（AppClassLoader 加载），但 Bootstrap ClassLoader 无法访问 classpath 里的实现类。线程上下文类加载器（TCCL）解决这个"加载器位于上层却需加载下层类"的困境。

```java
import java.sql.Driver;
import java.sql.DriverManager;
import java.util.ServiceLoader;

/**
 * 演示 JDBC 加载背后 SPI + TCCL 机制
 *
 * JDBC 4.0 起，DriverManager 通过 ServiceLoader 自动发现驱动
 * ServiceLoader.load() 默认使用当前线程的上下文类加载器
 */
public class JDBCSpiDemo {
    public static void main(String[] args) throws Exception {
        // 1. 被引导加载的接口: java.sql.Driver 是 Bootstrap 加载
        System.out.println("Driver class loader: "
            + Driver.class.getClassLoader());  // null

        // 2. TCCL 默认是 AppClassLoader
        Thread current = Thread.currentThread();
        System.out.println("TCCL: "
            + current.getContextClassLoader().getClass().getSimpleName());

        // 3. 显式切换 TCCL（容器场景常见：如 Tomcat、OSGi）
        ClassLoader customLoader = new PathClassLoader("/opt/app/lib");
        current.setContextClassLoader(customLoader);

        // 4. ServiceLoader 用 TCCL 加载实现类
        ServiceLoader<Driver> drivers = ServiceLoader.load(Driver.class);
        for (Driver d : drivers) {
            System.out.println("Found: " + d.getClass().getName()
                + " via " + d.getClass().getClassLoader());
        }
    }
}
```

### 观察类加载过程

使用 `-verbose:class` 参数打印每个类的加载顺序：

```bash
javac ClassLoaderHierarchy.java

# JDK 8
java -verbose:class ClassLoaderHierarchy 2>&1 | grep "ClassLoaderHierarchy\|Loaded.*from"

# 输出示例:
# [Opened C:\Program Files\Java\jdk1.8.0_201\lib\rt.jar]
# [Loaded java.lang.Object from rt.jar]
# [Loaded ClassLoaderHierarchy from file:/...]
```

关键观察：
- `java.lang.Object` 最先加载（Bootstrap ClassLoader）
- 目标类 `ClassLoaderHierarchy` 由 `AppClassLoader` 加载
- `loadClass` 优先委托父加载器，确认父加载器未找到才执行 `findClass`
