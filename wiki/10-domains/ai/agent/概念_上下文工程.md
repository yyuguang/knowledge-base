# 概念_上下文工程

## 定义

上下文工程（Context Engineering）是系统化设计、组装和管理 LLM 输入信息（上下文窗口内容）的方法论。不同于 prompt engineering 关注"怎么说"，context engineering 关注"给什么信息"。

## 核心挑战

| 挑战 | 描述 |
|------|------|
| **上下文窗口有限** | 即使 128K tokens 也有上限，信息需要筛选 |
| **上下文腐烂（Context Rot）** | 无关/错误信息在长上下文中导致模型注意力稀释 |
| **JIT 上下文** | 仅在需要时才注入信息，避免提前填充无用数据 |
| **Lost-in-Middle** | 模型对上下文中间部分信息关注度最低 |

## GSSC 四步流程

```
Gather  →  Select  →  Structure  →  Compress
收集      筛选       结构化        压缩
```

### 1. Gather（收集）

从多个来源汇聚候选信息：
- 记忆系统检索（Working / Episodic / Semantic）
- RAG 管道向量检索结果
- 工具调用的 Observation 历史
- 用户偏好和系统配置

### 2. Select（筛选）

```python
# 根据相关性和重要性筛选候选上下文
def select_context(candidates, query, max_items=5):
    scored = []
    for c in candidates:
        relevance = cosine_similarity(embed(c), embed(query))
        importance = c.get("importance", 0.5)
        recency = recency_score(c.get("timestamp"))
        # 融合评分
        score = relevance * 0.6 + importance * 0.2 + recency * 0.2
        scored.append((score, c))
    scored.sort(reverse=True)
    return [c for _, c in scored[:max_items]]
```

### 3. Structure（结构化）

按"先重要后次要，先整体后细节"组织信息：

```
[System Prompt] → [任务指令] → [高优先级上下文] → [补充信息] → [对话历史]
```

### 4. Compress（压缩）

对过长的信息进行摘要：

```python
def compress_if_needed(text, max_tokens):
    """如果文本超过 token 限制，调用 LLM 压缩"""
    if approx_tokens(text) <= max_tokens:
        return text
    return llm.invoke([
        {"role": "system", "content": "请用最简洁的语言概括以下内容的核心要点。"},
        {"role": "user", "content": text}
    ])
```

## 实际应用：Agent Loop 中的上下文工程

在 ReAct Agent 的每轮循环中，上下文工程体现在：

```python
# 1. Gather：从工具执行器和历史记录收集
tools_desc = self.tool_executor.getAvailableTools()
history_str = "\n".join(self.history)  # 累积的 Thought-Action-Observation

# 2. Structure：按模板组织
prompt = REACT_PROMPT_TEMPLATE.format(
    tools=tools_desc,      # 可用工具
    question=question,     # 原始问题
    history=history_str    # 交互历史
)

# 3. Compress（隐式）：oldest 历史可能被截断
# history 不无限增长 — 通过 max_steps 限制循环次数来控制
```

## JIT（Just-In-Time）上下文注入

核心理念：**不在循环开始前注入所有信息，而是在需要时才检索**。

赛博小镇中的实践：
- NPC 背景对话：批量预生成，缓存后按需注入
- 玩家交互：实时检索向量记忆 + 短期记忆拼装上下文

## 关键反模式

| 反模式 | 后果 | 修正 |
|--------|------|------|
| 上下文过载 | 模型忽略关键信息 | 严格筛选 ≤5 条最相关信息 |
| 上下文腐烂 | 早期错误污染后续推理 | 定期清理/重置历史 |
| 全量传输 | Token 浪费 | 用摘要替代原文 |
| 格式不一致 | 解析失败 | 统一 JSON/Markdown 结构化 |
