# 概念_垃圾收集算法

## 定义

垃圾收集算法是 JVM 自动管理内存的方法论，解决"哪些对象需要回收"和"如何回收"两大问题。

## 对象存活判定

- **引用计数法**：循环引用缺陷，主流 JVM 已弃用
- **可达性分析**：从 GC Roots（栈帧引用、静态变量、JNI 等）向下搜索，不可达即死亡
- **四种引用**：强引用（不回收）→ 软引用（OOM 前）→ 弱引用（下次 GC）→ 虚引用（仅通知）

### 四种引用实战演示

Java 提供 Reference 体系控制 GC 行为，不同引用级别决定了对象在 GC 中的"存活弹性"：

```java
import java.lang.ref.*;
import java.util.ArrayList;
import java.util.List;

public class ReferenceDemo {
    private static final int _1MB = 1024 * 1024;

    public static void main(String[] args) throws InterruptedException {
        // ---- 1. Strong Reference (默认) ----
        Object strong = new byte[10 * _1MB];
        System.gc();
        // strong 仍然可达，不会被回收

        // ---- 2. SoftReference — OOM 前才回收 ----
        SoftReference<byte[]> softRef = new SoftReference<>(new byte[10 * _1MB]);
        System.out.println("before OOM, soft: " + softRef.get());  // 非 null
        // 不断分配直到触发 OOM，在 OOM 前软引用会被回收
        try {
            List<byte[]> eater = new ArrayList<>();
            while (true) eater.add(new byte[10 * _1MB]);
        } catch (OutOfMemoryError e) {
            System.out.println("after OOM, soft: " + softRef.get());  // null
        }

        // ---- 3. WeakReference — 下次 GC 回收 ----
        Object weakObj = new Object();
        WeakReference<Object> weakRef = new WeakReference<>(weakObj);
        System.out.println("before GC, weak: " + weakRef.get());  // 非 null
        weakObj = null;
        System.gc();
        Thread.sleep(100);     // 等 GC 完成
        System.out.println("after GC, weak: " + weakRef.get());   // null

        // ---- 4. PhantomReference — 仅作为 GC 通知 ----
        ReferenceQueue<Object> rq = new ReferenceQueue<>();
        Object phantomObj = new Object();
        PhantomReference<Object> phantomRef = new PhantomReference<>(phantomObj, rq);
        phantomObj = null;
        System.gc();
        Thread.sleep(100);
        System.out.println("phantom get: " + phantomRef.get());           // always null
        System.out.println("enqueued?: " + (rq.poll() != null));          // true
    }
}
```

**选择指南**：
| 引用类型 | 回收时机 | 典型用途 |
|----------|----------|----------|
| 强引用 | 永不回收（GC Roots 可达） | 普通对象引用 |
| 软引用 SoftRef | OOM 前一次 GC | 图片/文档缓存 |
| 弱引用 WeakRef | 下次 GC 无论内存是否够 | WeakHashMap、ThreadLocal |
| 虚引用 PhantomRef | GC 后入队列（get 永远 null） | NIO DirectBuffer 释放通知 |

### GC Roots 枚举实战

可达性分析的起点是 GC Roots。以下代码展示了不同 GC Root 类型的来源：

```java
public class GCRootsDemo {
    // 1. 静态字段 → GC Root
    private static Object staticField = new Object();     // 作为 GC Root

    // 2. 常量 → GC Root（在运行时常量池中）
    private static final Object CONSTANT = new Object();  // 作为 GC Root

    public static void main(String[] args) {
        // 3. 栈帧局部变量 → GC Root（线程存活期间）
        Object stackLocal = new Object();                 // 作为 GC Root

        // 4. JNI 全局引用 → GC Root（Native 方法持有）
        // 无纯 Java 示例，调用 Thread 的 native 方法时 JNI 会持有引用

        // 5. 被 synchronized 持有的对象 → GC Root
        synchronized (stackLocal) {
            // lock 期间 stackLocal 作为 GC Root
        }

        // 6. 活跃线程本身 → GC Root
        new Thread(() -> {
            Object threadLocal = new Object();  // 线程运行期间为 GC Root
            try { Thread.sleep(10000); } catch (InterruptedException ignored) {}
        }).start();

        // 用 jmap 可导出 GC Roots:
        // jmap -dump:live,format=b,file=heap.hprof <pid>
        // 然后用 MAT 的 "GC Roots" 视图查看所有 GC Root 链
    }
}
```

### finalize() 自我救赎

`finalize()` 已被 JDK 9 标记为 `@Deprecated`，但理解其机制有助于掌握对象生命周期。对象在 GC 时可趁机"复活"一次：

```java
public class FinalizeEscapeGC {
    public static FinalizeEscapeGC SAVE_HOOK = null;

    public void isAlive() {
        System.out.println("yes, I am still alive :)");
    }

    @Override
    protected void finalize() throws Throwable {
        super.finalize();
        // 自我救赎：把 this 重新赋给 GC Root 静态变量
        SAVE_HOOK = this;
        System.out.println("finalize() executed, self-rescue!");
    }

    public static void main(String[] args) throws Exception {
        SAVE_HOOK = new FinalizeEscapeGC();

        // 第一次：置 null → GC → finalize 救回
        SAVE_HOOK = null;
        System.gc();
        Thread.sleep(500);  // finalizer 线程优先级低，等待执行
        if (SAVE_HOOK != null) SAVE_HOOK.isAlive();   // 输出 alive
        else System.out.println("dead");

        // 第二次：finalize() 只会被 JVM 调用一次，这次救不回来了
        SAVE_HOOK = null;
        System.gc();
        Thread.sleep(500);
        if (SAVE_HOOK != null) SAVE_HOOK.isAlive();
        else System.out.println("dead");              // 输出 dead
    }
}
```

**机制说明**：JVM 对任何对象的 `finalize()` 最多只调用一次。对象第一次 GC 时若 `finalize()` 尚未执行且重写了该方法，则放入 F-Queue 由 Finalizer 线程异步执行。执行期间对象可"复活"。第二次 GC 时 `finalize()` 不再执行，对象直接回收。

### ThreadLocal 内存泄漏演示

ThreadLocal 使用弱引用（ThreadLocalMap 的 Entry extends WeakReference），当 ThreadLocal 实例被 GC 后，Entry 的 key 变为 null，但 value 仍被线程持有。配合线程池（线程复用不回收）极易引发内存泄漏：

```java
// -Xmx50m -Xms50m
import java.util.concurrent.*;

public class ThreadLocalOOM {
    static class BigObject {
        private byte[] data = new byte[1024 * 1024]; // 1MB
    }

    public static void main(String[] args) throws Exception {
        ExecutorService pool = Executors.newFixedThreadPool(5);

        for (int i = 0; i < 500; i++) {
            final int idx = i;
            pool.execute(() -> {
                ThreadLocal<BigObject> tl = new ThreadLocal<>();
                tl.set(new BigObject());  // 1MB value 存入线程的 ThreadLocalMap
                // ⚠️ 未调用 tl.remove()！
            });
            // tl 引用离开 scope 后，ThreadLocal 对象本身（key）很快被 GC
            // 但线程池中的线程不会回收，ThreadLocalMap 的 Entry.value 强引用 BigObject
            // 导致 1MB 永久无法释放 → 反复提交任务最终 OOM
        }

        // 观察 jstat 可以看到老年代持续增长直到 OOM
        // jstat -gc <pid> 1000
        pool.shutdown();
        pool.awaitTermination(1, TimeUnit.MINUTES);
    }
}
```

**修复方式**：每次使用完 ThreadLocal 后必须调用 `tl.remove()`，或使用 `try-finally` 保证清理：

```java
ThreadLocal<BigObject> tl = new ThreadLocal<>();
try {
    tl.set(new BigObject());
    // ... 业务逻辑
} finally {
    tl.remove();  // 必做！
}
```

## 基本算法

| 算法 | 描述 | 优缺点 |
|------|------|--------|
| **标记-清除** | 标记存活→清除垃圾 | 内存碎片化，分配效率下降 |
| **标记-复制** | 将存活对象复制到另一半 | 效率高但浪费空间。HotSpot 新生代默认 Eden:Survivor=8:1 |
| **标记-整理** | 标记存活→向一端移动 | 无碎片，但移动开销大，需 STW |

## 使用场景

- **新生代**：标记-复制（对象朝生夕灭，复制成本低）
- **老年代**：标记-清除（CMS）或标记-整理（Parallel Old/G1）

## 分代收集理论

- **弱分代假说**：绝大多数对象朝生夕灭
- **跨代引用**：通过**记忆集（Remembered Set）+ 卡表（Card Table）**解决
- **写屏障**：在引用赋值时维护卡表状态
- **伪共享（False Sharing）**：卡表更新时的并发性能问题

## 并发标记的核心挑战

- **三色标记**：白色（未访问）/ 灰色（访问中）/ 黑色（已扫描）
- **对象消失问题**：黑色对象插新引用指向白色 + 灰色对象删除到白色的引用
- **解决方案**：增量更新（CMS，记录新插入引用）vs 原始快照 SATB（G1/Shenandoah，记录删除的引用）

## 在本知识库中的应用

- 理解 GC 日志时：`[ParNew: 803142K->100352K(903168K)]` 的回收效率与算法选择直接相关
- [[concepts/java/jvm/概念_垃圾收集器]] 是算法的具体实践者，选型需结合算法特性
- JVM 调优中 GC 停顿分析需掌握三色标记和 SATB/增量更新的区别
