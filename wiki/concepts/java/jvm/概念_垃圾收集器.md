# 概念_垃圾收集器

## 定义

垃圾收集器是垃圾收集算法的具体实践者。《Java虚拟机规范》对收集器如何实现无强制规定，各厂商的虚拟机实现各有差异。HotSpot 提供多款收集器以适应不同场景。

## 完整收集器组合与启动参数

下表列出 HotSpot JDK 8+ 中所有主流的收集器配对及其 JVM 启动参数，便于选型和线上快速切换：

| 组合（新生代+老年代）             | JVM 启动参数                                                    | 适用场景               |
|-------------------------------|----------------------------------------------------------------|------------------------|
| Serial + Serial Old            | `-XX:+UseSerialGC`                                             | 客户端/单核/小堆       |
| ParNew + CMS                   | `-XX:+UseConcMarkSweepGC -XX:+UseParNewGC`                     | 低延迟 Web 应用         |
| Parallel Scavenge + Parallel Old | `-XX:+UseParallelGC -XX:+UseParallelOldGC`                    | 吞吐量优先批处理       |
| Serial (新生代) + CMS (老年代)   | `-XX:+UseConcMarkSweepGC -XX:-UseParNewGC`                     | CMS 退化后备           |
| G1                             | `-XX:+UseG1GC`                                                 | 大堆（>4G）低延迟      |
| Shenandoah                     | `-XX:+UseShenandoahGC`                                         | 超低延迟大堆（JDK12+） |
| ZGC                            | `-XX:+UseZGC`                                                  | 亚毫秒级停顿（JDK11+） |
| Epsilon                        | `-XX:+UnlockExperimentalVMOptions -XX:+UseEpsilonGC`            | 性能测试/不回收        |

**常用调优参数速查**：
```
# Parallel GC 吞吐量调优
-XX:MaxGCPauseMillis=200         # 最大 GC 停顿目标
-XX:GCTimeRatio=19               # GC 时间占比 = 1/(1+19) = 5%

# CMS 关键参数
-XX:CMSInitiatingOccupancyFraction=75   # 老年代占 75% 触发 CMS
-XX:+UseCMSInitiatingOccupancyOnly      # 仅用此阈值（禁用 JVM 自行估算）

# G1 关键参数
-XX:MaxGCPauseMillis=200               # 停顿目标
-XX:G1HeapRegionSize=4m                # Region 大小（1-32M，2的幂）
-XX:InitiatingHeapOccupancyPercent=45  # 堆占用达 45% 触发并发标记

# ZGC 关键参数
-XX:+UseZGC
-XX:ZAllocationSpikeTolerance=2        # 分配突增容忍度
-XX:+ZUncommit                         # 归还未使用内存给 OS
-XX:ZUncommitDelay=300                 # 300s 后才归还
```

## 经典收集器（JDK 7–10）

### 新生代

- **Serial**：单线程，STW，客户端模式默认，简单高效
- **ParNew**：Serial 的多线程版，只能配合 CMS
- **Parallel Scavenge**：吞吐量优先，自适应调节（GC Ergonomics），-XX:MaxGCPauseMillis / -XX:GCTimeRatio

### 老年代

- **Serial Old**：单线程，标记-整理，CMS 后备预案
- **Parallel Old**：Parallel Scavenge 的老年代搭档，多线程标记-整理
- **CMS（Concurrent Mark Sweep）**：低延迟，6 阶段执行（初始标记→并发标记→重新标记→并发清除→重置），增量更新解决并发标记，Concurrent Mode Failure 退化 Serial Old

#### CMS GC 日志深度解析

下面是一段真实的 CMS GC 日志（JDK 8），添加了逐行注释说明每个阶段的作用：

```bash
# JVM 参数: -XX:+UseConcMarkSweepGC -Xlog:gc* -verbose:gc

# === Phase 1: 初始标记 (Initial Mark) ===
# 仅标记 GC Roots 直接关联的对象 → STW，但很快
[GC (CMS Initial Mark) [1 CMS-initial-mark: 0K(4096000K)] 103427K(4718592K), 0.0071869 secs]

# === Phase 2: 并发标记 (Concurrent Mark) ===
# 从 GC Roots 开始遍历对象图，与应用线程并发 → 无 STW
[CMS-concurrent-mark-start]
[CMS-concurrent-mark: 0.326/0.326 secs]
# ↳ CPU time: 0.326s / Wall time: 0.326s

# === Phase 3: 并发预清理 (Concurrent Preclean) ===
# 处理并发标记期间发生变化的引用 → 无 STW
[CMS-concurrent-preclean-start]
[CMS-concurrent-preclean: 0.020/0.022 secs]

# === Phase 4: 可中断的并发预清理 (Abortable Preclean) ===
# 等待 Young GC 发生以减小下一阶段的标记量；可打断
[CMS-concurrent-abortable-preclean-start]
[CMS-concurrent-abortable-preclean: 0.442/0.442 secs]

# === Phase 5: 重新标记 (Final Remark / Remark) ===
# 修正并发标记期间的引用变化（增量更新/卡表）→ STW，通常几十~几百 ms
[GC (CMS Final Remark) [YG occupancy: 803142K(903168K)]
[Rescan (parallel), 0.0913220 secs]                # 并行扫描卡表
[weak refs processing, 0.0001320 secs]             # 处理弱引用
[class unloading, 0.0010650 secs]                  # 卸载类
[scrub symbol table, 0.0007550 secs]               # 清理符号表
[scrub string table, 0.0002120 secs]               # 清理字符串表
[CMS-remark: 387429K(4096000K)] 1190571K(4718592K), 0.0935399 secs]

# === Phase 6: 并发清除 (Concurrent Sweep) ===
# 清除未标记（即垃圾）对象，与应用线程并发 → 无 STW
[CMS-concurrent-sweep-start]
[CMS-concurrent-sweep: 0.683/0.685 secs]

# === Phase 7: 重置 (Concurrent Reset) ===
# 清理内部状态，准备下一个 CMS 周期
[CMS-concurrent-reset-start]
[CMS-concurrent-reset: 0.011/0.011 secs]

# ==========================================
# ⚠️ Concurrent Mode Failure 日志示例：
# ==========================================
# [GC (CMS Initial Mark) [1 CMS-initial-mark: 3987456K(4096000K)] ...]
# [CMS-concurrent-mark: 1.234/1.234 secs]
# [CMS-concurrent-preclean: 0.020/0.022 secs]
# [CMS-concurrent-abortable-preclean: 8.442/8.442 secs]
# -- 并发预清理阶段耗时过长，老年代在并发清理完成前被填满 --
# [GC (Allocation Failure) [CMS (concurrent mode failure): 4096000K->4096000K ...
#  接下来进行 Serial Old 标记-整理，即 Full GC → 长时间 STW
```

**日志关注点**：
- CMS-initial-mark STW 时间 → 通常 <10ms 正常
- CMS-remark STW 时间 → 若 >500ms 应检查晋升过快
- concurrent mode failure → 说明老年代填充速度快过 CMS 清理速度，升级到 G1 或增大堆
- promotion failed → 老年代碎片化 "过早" 触发，同样会退化为 Full GC

### G1（Garbage First）

- Region 化堆（1MB–32MB），新生代/老年代不再物理连续
- **RSet**：每个 Region 记录谁引用自己，避免全堆扫描
- 停顿可预测（-XX:MaxGCPauseMillis），基于回收集合（CSet）和 SATB
- **JDK 9 起成为默认收集器**

#### G1 GC 日志解析

G1 的核心是将堆切分为 Region 并基于 CSet（Collection Set）选择性回收。日志中 Young GC 和 Mixed GC 有不同特征：

```bash
# JVM 参数: -XX:+UseG1GC -Xlog:gc* -verbose:gc

# === Young GC (G1 Evacuation Pause - young) ===
# 回收 Eden + Survivor Region → STW，通常 <100ms
[GC pause (G1 Evacuation Pause) (young), 0.0220152 secs]
   [Parallel Time: 21.2 ms, GC Workers: 8]         # 8 个 GC 线程并行
      [GC Worker Start (ms):  ...]                 # 各线程开始时间
      [Ext Root Scanning (ms):  1.2]               # GC Roots 扫描
      [Update RS (ms):          3.5]               # 更新 Remembered Set
      [Scan RS (ms):            2.1]               # 扫描 RSet
      [Object Copy (ms):        14.1]              # 复制存活对象
      [Termination (ms):        0.2]               # 线程终止
   [Code Root Fixup: 0.1 ms]                       # 修正代码根
   [Clear CT: 0.2 ms]                              # 清理卡表
   [Eden: 312.0M(312.0M)->0.0B(312.0M)             # Eden 清空
    Survivors: 16.0M->16.0M                          # Survivor 不变
    Heap: 412.0M(2.0G)->124.3M(2.0G)]              # 堆从 412M 降到 124M
 [Times: user=0.14 sys=0.00, real=0.02 secs]       # STW 仅 20ms

# === Initial Mark (Young GC 附带) ===
# G1 的初始标记 piggyback 在 Young GC 上，几乎无额外 STW
[GC pause (G1 Evacuation Pause) (young) (initial-mark), 0.0251123 secs]
   ... (类似 young GC，但多了 initial-mark)

# === 并发标记 ===
[GC concurrent-root-region-scan-start]             # 扫描根 Region
[GC concurrent-root-region-scan-end, 0.0012345 secs]
[GC concurrent-mark-start]                         # 并发标记开始
[GC concurrent-mark-end, 0.2345678 secs]           # 并发标记结束

# === 重新标记 (Remark) ===
# STW，修正并发标记期间引用变化（SATB）
[GC remark, 0.0123456 secs]

# === 清理 (Cleanup) ===
# STW，计算回收价值，构建 Mixed GC 候选 CSet
[GC cleanup, 0.0012345 secs]

# === Mixed GC (G1 Evacuation Pause - mixed) ===
# 回收所有年轻代 + 部分老年代 Region → 这才是 G1 的 "增量回收"
[GC pause (G1 Evacuation Pause) (mixed), 0.0456789 secs]
   [Eden: 312.0M(312.0M)->0.0B(312.0M)
    Survivors: 16.0M->16.0M
    Heap: 856.3M(2.0G)->534.7M(2.0G)]             # 老年代也被回收了一部分

# === ⚠️ Full GC (G1 后备方案) ===
# 当 Mixed GC 来不及回收、堆用尽时触发：单线程，标记-整理，STW 长
[Full GC (Allocation Failure) 1987M->1023M(2.0G), 2.3456789 secs]
[Eden: 0.0B(312.0M)->0.0B(312.0M)
 Survivors: 0.0B->0.0B
 Heap: 1987M(2.0G)->1023M(2.0G)]
 [Times: user=5.23 sys=0.12, real=2.35 secs]      # ⚠️ 停顿 2.3s
```

**G1 日志关键指标**：
- Young GC 停顿是否稳定在 `MaxGCPauseMillis` 附近
- 是否出现 Full GC（G1 的 Full GC 代价极大，应避免）
- Mixed GC 出现的频率和回收效果（老年代回收了多少）
- `To-space exhausted` → 晋升失败，老年代空间不足

#### G1 大对象（Humongous Object）特殊处理

G1 中对象大小 **超过 Region 一半** 即判定为大对象（Humongous），分配策略与普通对象完全不同：

```java
// -XX:+UseG1GC -XX:G1HeapRegionSize=4m -Xmx256m -Xlog:gc*
public class G1HumongousDemo {
    private static final int _2MB = 2 * 1024 * 1024;

    public static void main(String[] args) {
        // Region 大小 4MB → 对象 > 2MB 即为 Humongous
        // 该对象分配在连续的 Region 中（Humongous Region）
        byte[] humongous = new byte[_2MB + 1024];  // > 2MB → Humongous

        System.out.println("humongous object allocated");

        // 再分配一个大对象观察 GC 日志
        byte[] another = new byte[_2MB + 1024];

        System.out.println("done, check GC log");
    }
}
```

**对应 GC 日志**：
```
[GC pause (G1 Humongous Allocation) (young), 0.0123456 secs]
   [G1 Humongous Allocation: 2.5M]     # 大对象分配触发的 GC
   ...
```

**Humongous 的特殊行为**：
1. **跳过 Survivor**：大对象不进入新生代 Survivor，直接分配到老年代 Humongous Region
2. **需连续 Region**：若对象大小跨越 N 个 Region，需找到 N 个连续的 Region
3. **仅 Full GC 回收**：Humongous 对象不在 Young/Mixed GC 中回收（G1 早期版本），容易导致碎片化
4. **实战建议**：避免频繁创建 > Region/2 大小的短命对象；增大 Region Size（`-XX:G1HeapRegionSize=8m`）或避免此类模式

## 低延迟收集器（JDK 11+）

| 收集器 | 核心原理 | 特点 |
|--------|----------|------|
| **Shenandoah** | 连接矩阵 + 并发回收 + 读屏障 | 停顿与堆大小无关 |
| **ZGC** | 染色指针（Colored Pointer）+ 读屏障 + Region 动态大小 | 几乎无停顿，亚毫秒级 |
| **Epsilon** | 无操作 | 不回收，仅用于性能测试 |

### ZGC 调优实战

ZGC（JDK 11 实验性，JDK 15 生产可用）以染色指针和读屏障实现亚毫秒级停顿。下面是一个完整的 ZGC 启动配置和基本测试：

```java
/*
 * ZGC 启动参数：
 * -XX:+UseZGC
 * -Xmx4g -Xms4g
 * -Xlog:gc*                     # JDK 11-16 日志
 * -Xlog:gc*=info                # JDK 17+ unified logging
 * -XX:+ZUncommit                # 归还未使用内存给 OS
 * -XX:ZUncommitDelay=300        # 闲置 5 分钟后归还
 */
public class ZGCDemo {
    public static void main(String[] args) throws Exception {
        // 持续分配 + 较少引用制造 GC 压力
        for (int i = 0; i < 10; i++) {
            byte[] arr = new byte[50 * 1024 * 1024];  // 50MB
            System.out.println("allocated 50MB, iter " + i);
            Thread.sleep(1000);
        }
        System.out.println("done, check ZGC pause times");
    }
}
```

**ZGC 日志关键特征**（JDK 17+）：
```
[0.350s][info][gc] GC(0) Garbage Collection (Warmup)
[0.350s][info][gc,start    ] GC(0) Pause Mark Start 0.024ms   # 亚毫秒
[0.354s][info][gc,phases   ] GC(0) Pause Mark End 0.032ms     # <1ms
[0.354s][info][gc,phases   ] GC(0) Concurrent Mark 2.112ms
[0.356s][info][gc,phases   ] GC(0) Concurrent Mark Continue 1.834ms
[0.358s][info][gc,phases   ] GC(0) Pause Mark End 0.021ms     # 亚毫秒
[0.360s][info][gc,phases   ] GC(0) Concurrent Process Non-Strong References 1.234ms
[0.362s][info][gc,phases   ] GC(0) Concurrent Relocate 1.045ms  # 并发迁移
[0.363s][info][gc,load     ] GC(0) Load: 6.28M/6.28M/6.28M    # 染色指针负载
```

**ZGC 关键优势**：
- Pause Mark Start/Mark End 均由读屏障自愈，停顿不受堆大小影响
- 染色指针将 GC 状态编码在指针中（object virtual address 的 42 bits 中），无需额外内存
- Concurrent Relocate 阶段可并发迁移存活对象，应用线程同步运行

## 使用场景（选型要点）

- **延迟敏感** → CMS / G1 / ZGC / Shenandoah
- **吞吐量优先** → Parallel Scavenge + Parallel Old
- **小内存/客户端** → Serial

**选型决策树**：

```
堆大小 < 100MB?  → Serial（简单高效）
     │
堆大小 100MB–4GB?  → Parallel（批处理）/ CMS（延迟敏感）
     │
堆大小 > 4GB?  → G1（JDK 9+, 通用首选）
     │
停顿 < 10ms?  → ZGC（JDK 15+, 染色指针）/ Shenandoah（JDK 12+）
```
