# 概念_类加载机制

## 定义

Java 虚拟机把描述类的数据从 Class 文件加载到内存，并对数据进行校验、转换解析和初始化，最终形成可被虚拟机直接使用的 Java 类型。与 C/C++ 不同，Java 的类型加载、连接和初始化均在运行期完成，这为 Java 提供了极高的扩展性和灵活性。

## 生命周期 7 阶段

加载（Loading）→ 验证（Verification）→ 准备（Preparation）→ 解析（Resolution）→ 初始化（Initialization）→ 使用（Using）→ 卸载（Unloading）

### 前 5 阶段详解
1. **加载**：通过全限定名获取二进制字节流 → 转为方法区数据结构 → 堆中生成 Class 对象
2. **验证**：文件格式 → 元数据 → 字节码（StackMapTable）→ 符号引用
3. **准备**：为 static 变量分配内存并设零值（final static 直接赋初始值）
4. **解析**：符号引用 → 直接引用（类/接口/字段/方法/方法句柄/方法类型/调用点限定符）
5. **初始化**：执行 `<clinit>`，合并所有 static 赋值 + static 块，JVM 保证线程安全

## 6 种主动初始化触发

- `new` / `getstatic` / `putstatic` / `invokestatic` 指令
- `java.lang.reflect` 反射
- 子类初始化触发父类
- JVM 启动时指定 main 类
- `java.lang.invoke.MethodHandle` 解析 REF_getStatic 等
- 默认方法（JDK 8+）

## 类加载器与双亲委派模型

- **Bootstrap ClassLoader** → **Extension ClassLoader** → **Application ClassLoader** → **自定义 ClassLoader**
- **优点**：基础类统一（如 java.lang.Object 保证一致），避免重复加载
- **破坏场景**：JDK 1.2 兼容、SPI（线程上下文类加载器）、OSGi（网状结构）
- **JDK 9 模块化（Jigsaw）**：类加载器结构变化，平台类加载器取代 Extension

## 在本知识库中的应用

- 排查 `ClassNotFoundException` / `NoSuchMethodError`：双亲委派机制导致 jar 版本冲突
- Tomcat 通过自定义 ClassLoader 实现 Web 应用隔离
