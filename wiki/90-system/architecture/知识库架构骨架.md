# 知识库架构骨架

## 当前结论

本知识库采用“**稳定归属优先，知识形态由元数据表达**”的结构。

目录只回答“这篇内容未来最稳定地住在哪里”；`type`、`status`、`sources` 等 frontmatter 回答“它是什么、来自哪里、成熟到什么程度”。因此，新增一种笔记形态或自动化能力时，通常不需要再新增一个顶层内容目录。

本架构保留 `raw/` 为只读来源层。2026-07-19 已完成所有 Wiki 页面的迁移并删除旧的类型优先目录；全部内容都位于本页定义的当前目录中。

## 目录职责与写入边界

| 位置 | 放什么 | 不放什么 | 典型产物 |
| --- | --- | --- | --- |
| `00-inbox` | 快速笔记、未分类摘录、待处理问题 | 已成熟的长期知识 | 每日输入、待处理清单 |
| `10-domains` | 不依赖某个单一项目的可复用知识 | 会议纪要、项目进度 | JVM、Agent、Spring 的机制与方法 |
| `20-projects` | 与目标、代码库、决策或实验绑定的材料 | 泛化后的通用概念 | 架构地图、ADR、实验记录、项目索引 |
| `30-sources` | 一份书、论文、课程、仓库“讲了什么” | 从多份资料提炼出的结论 | 来源摘要、阅读笔记、证据入口 |
| `40-maps` | 多个页面之间的导航、比较、路线和问题框架 | 单一来源的线性摘要 | 学习路径、技术选型、主题地图 |
| `90-system` | 模板、工作流、查询、脚本和架构规范 | 领域知识正文 | 自动化说明、Dataview 查询、模板 |
| `_meta` | 对整个库生效的登记与迁移信息 | 普通学习笔记 | 迁移说明、目录登记 |

## 归档决策树

```text
新内容
├─ 还不确定长期归属？ → 00-inbox/
├─ 核心是“这份资料说了什么”？ → 30-sources/
├─ 服务于一个有明确目标的项目？ → 20-projects/<项目>/
├─ 一个月后仍可脱离项目复用？ → 10-domains/<领域>/
├─ 主要用于把多页串成入口、比较或路线？ → 40-maps/
└─ 用于管理或扩展知识库本身？ → 90-system/
```

如果一篇内容同时符合两个位置，选择其**主要维护语境**：

- 在项目中产生、仍依赖项目细节：先放项目；成熟后在领域中写出“去项目化”的独立页面，并互相链接。
- 资料与知识结论并存：资料事实留在 `30-sources`，跨来源的结论写入 `10-domains` 或 `40-maps`。
- 不能通过复制解决多归属问题；使用路径完整的 wikilink 建立第二入口。

## 页面命名

- 普通知识页：使用清晰、稳定的中文主题名，如 `JVM内存区域.md`、`ReAct模式.md`。
- 项目决策：`ADR_YYYYMMDD_决策主题.md`。
- 实验记录：`实验_主题.md`。
- 来源摘要：`来源_资料标题.md`。
- 导航页：领域和项目使用 `_index.md`；其他地图使用 `主题_名称.md` 或 `路线_名称.md`。

文件名不再强制添加“概念_”“实体_”“综述_”等类型前缀；类型由 frontmatter 表达。

## Frontmatter 规范

所有新页面保留旧库兼容的 `type`，并新增 `area` 与 `scope`。

```yaml
---
type: concept                  # source | concept | entity | comparison | overview | insight | decision | experiment | map
area: java/jvm                 # 稳定领域；允许为空
scope: durable                 # inbox | durable | project | source | map | system
project: []                    # 仅项目相关页面填写，例如 [mall-swarm]
summary: "一句话说明页面能解决的问题"
sources: []                    # raw 路径或来源摘要页的完整 wikilink
aliases: []
tags: []
status: draft                  # draft | evolving | stable | archived
confidence: 0.0
created: "YYYY-MM-DD HH:mm:ss"
updated: "YYYY-MM-DD HH:mm:ss"
---
```

约定：

- `area` 是检索维度，不是强制目录镜像；但通常和 `10-domains` 路径一致。
- `scope` 必须与目录相符：`10-domains` 使用 `durable`，`20-projects` 使用 `project`，依此类推。
- `type` 可以变化，目录通常不变。例如一页项目 ADR 仍放在项目内，而不是新建 `decisions/` 顶层目录。
- 对未经原始材料或运行时验证的推断，降低 `confidence` 并在正文标出待验证点。

## 导航与链接规则

- 每个活跃领域和项目必须有一个 `_index.md`，它是导航页而不是知识内容的复制品。
- 普通页面尽可能提供至少两个**有语义的**内部链接；来源页可例外。
- 使用完整路径，例如 `[[10-domains/java/_index]]`、`[[10-domains/java/jvm/概念_JVM内存区域]]`。
- 页面统一使用当前目录的完整路径；避免依赖旧的类型优先路径。
- `wiki/index.md` 只保留全库入口，不承担枚举每一页的职责。

## 新功能的落点

| 想新增的能力 | 应放位置 | 说明 |
| --- | --- | --- |
| 日记、快速捕获 | `00-inbox/` | 按需清理或分流，不提前分类 |
| 学习路线、面试题图谱 | `40-maps/` | 连接既有知识，不复制正文 |
| Dataview / 搜索视图 | `90-system/queries/` | 查询是能力，不是知识类型 |
| Agent 提示词、导入流程 | `90-system/workflows/` | 记录可重复的工作方式 |
| 页面模板 | `90-system/templates/` | 用于生成一致的新页 |
| 自动化脚本 | `90-system/scripts/` | 与可执行说明相邻维护 |
| 新项目 | `20-projects/<项目>/` | 项目结束后沉淀可复用内容到领域 |

## 维护检查

新建或更新页面前，确认：

1. 它是否放在最稳定的维护语境？
2. 它的 `scope` 与目录是否一致？
3. 是否已有同主题页面可更新，避免重复？
4. 是否应链接一个来源页和一个相关知识页？
5. 是否把“资料事实”“推断”“待验证项”明确区分？
6. 是否需要更新对应 `_index.md` 或 `40-maps`？
