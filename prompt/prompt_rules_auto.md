# Obsidian 知识库 Agent 提示词模板
## 1. 导入新资料：Ingest 模式

适合你把文章、笔记、网页、PDF 摘要放进知识库。

```
请对以下资料执行 Ingest。

要求：
1. 判断所属领域 area
2. 创建或更新 source page
3. 提取重要概念，并创建或更新 concept page
4. 如果能形成主题理解，更新 overview page
5. 更新 wiki/index.md
6. 更新 wiki/log.md
7. 不要只做摘要，要把内容编译成未来可复用的知识
8. 有详细的excalidraw流程图与架构图

资料如下：

<粘贴资料内容>
```

---

## 2. 基于 raw 文件导入

适合你已经把原文放到 `raw/` 目录。

```
请基于 raw/<文件名> 执行 Ingest。

要求：
1. raw 文件只读，不要修改
2. 生成或更新 wiki/sources/<area>/来源_xxx.md
3. 提取 1–3 个最重要的概念，更新到 wiki/concepts/<area>/
4. 如果已有相关页面，优先更新，不要重复创建
5. 更新 wiki/index.md 和 wiki/log.md
6. 输出最终修改清单
```

---

## 3. 强制高质量导入

当你发现 Agent 又开始偷懒，只输出几条 bullet，用这个。

```
请重新执行 Ingest，但这次不要做极简摘要。

质量要求：
1. source page 至少包含：
   - 一句话总结
   - 来源信息
   - 5–10 个核心观点
   - 关键论证
   - 重要概念
   - 我的理解
   - 未解决问题
   - 相关链接

2. concept page 不能只写概念名，必须包含：
   - 定义
   - 为什么重要
   - 使用场景
   - 核心机制
   - 例子
   - 常见误区
   - 和其它概念的关系
   - 来源

3. 最终产物要做到：
   - 一个月后仍然有用
   - 未来可以直接基于 wiki 回答问题
   - 不需要重新阅读 raw 原文
```

---

## 4. 查询知识库：Query 模式

适合你问问题，但希望它优先用自己的 wiki。

```
请基于 wiki 回答这个问题：

<你的问题>

要求：
1. 先查 wiki/index.md
2. 再查相关 area 下的 concept/source/overview 页面
3. 只使用相关页面，不要全库扫描
4. 如果 wiki 内容不足，请明确指出缺口
5. 如果这次回答形成了可复用知识，请写回 wiki
```

示例：

```
请基于 wiki 回答这个问题：

Spring AI 和 LangChain4j 的区别是什么？

要求：
1. 先查 wiki/index.md
2. 再查 java/spring-ai 和 java/langchain4j 相关页面
3. 输出对比结论
4. 如果对比内容有复用价值，请创建或更新 comparison 页面
```

## 5. 更新已有页面：Update 模式

适合你让它改某个概念页。

```
请更新这个 wiki 页面：

wiki/concepts/<area>/概念_xxx.md

更新目标：
<说明你想补充什么>

要求：
1. 先读取目标页面
2. 再检查同 area 下是否有重复或相似页面
3. 只补充缺失内容
4. 如果页面结构太差，可以重构
5. 保留已有有效内容
6. 更新 wiki/log.md
7. 输出变更摘要
```

---

## 6. 把一段新理解补进知识库

这个很适合你学习时突然有一个想法。

```
我有一段新的理解，请判断应该写入哪个 wiki 页面。

新理解：

<粘贴你的理解>

要求：
1. 判断这是 concept、overview、comparison 还是 insight
2. 先搜索是否已有相关页面
3. 如果已有页面，更新已有页面
4. 如果没有，创建新页面
5. 添加必要的上下文链接
6. 更新 wiki/index.md 和 wiki/log.md
```

---

## 7. 生成主题综述：Synthesis 模式

适合多个资料看完之后，让它整理成自己的理解。

```
请对主题「<主题名>」执行 Synthesis。

要求：
1. 搜索相关 sources、concepts、comparisons、overview
2. 找出重复出现的模式
3. 找出不同资料之间的冲突或差异
4. 形成当前阶段的总体理解
5. 更新或创建 overview 页面
6. 不要只总结资料，要形成我的知识库中的当前判断
7. 更新 wiki/index.md 和 wiki/log.md
```

示例：

```
请对主题「LLM 知识库设计」执行 Synthesis。

要求：
1. 搜索 ai/agent、knowledge-management 相关页面
2. 找出 Karpathy LLM Wiki 思路和我当前知识库设计的关系
3. 形成一个 overview
4. 输出当前判断、核心概念、未解决问题、下一步优化方向
```

## 8. 对比两个概念：Comparison 模式

```
请基于 wiki 对比以下两个概念：

A：<概念A>
B：<概念B>

要求：
1. 先搜索已有 concept 页面
2. 如果已有 comparison 页面，更新它
3. 如果没有，则创建 wiki/comparisons/<area>/<A>_vs_<B>.md
4. 对比维度包括：
   - 定义
   - 目标
   - 使用场景
   - 成本
   - 优点
   - 局限
   - 选择建议
5. 更新 index 和 log
```

---

## 9. 检查知识库质量：Lint 模式

```
请对当前 wiki 执行 Lint。

检查范围：
1. 重复页面
2. 孤立页面
3. 缺少 frontmatter 的页面
4. 定义太浅的 concept 页面
5. 缺少 overview 的主题
6. 链接没有上下文说明的问题
7. index 和实际页面不一致的问题

要求：
1. 不要直接修改
2. 输出问题列表
3. 每个问题给出修改建议
4. 最多列出 10 个最重要的问题
```

---

## 10. 修复 Lint 问题

当它已经给出问题后，用这个。

```
请根据刚才的 Lint 结果执行修复。

修复范围：
<指定要修复的问题编号>

要求：
1. 只修复我指定的问题
2. 不要做额外大规模重构
3. 修改前先检查重复页面
4. 更新相关页面、index 和 log
5. 输出变更清单
```

---

## 11. 防止生成垃圾页面

这个可以加在任何提示词后面。

```
质量约束：
1. 不要创建只有几行内容的浅页面
2. 不要为了链接而链接
3. 不要重复创建语义相同的页面
4. 如果内容不足以支撑新页面，就先写入已有页面的「待补充」部分
5. 每个新页面必须有明确的长期用途
```

---

## 12. 防止它只摘要不沉淀

这个也可以加在 Ingest 后面。

```
注意：
这不是摘要任务，而是知识库编译任务。

请把资料转化为：
1. 可复用概念
2. 可查询结构
3. 可链接页面
4. 我的理解
5. 后续可演化的问题

不要只复述原文。
```

---

## 13. 让它输出修改计划，而不是直接改

适合大改前使用。

```
请先不要修改文件。

请针对以下目标给出 wiki 更新计划：

目标：
<你的目标>

要求：
1. 需要新增哪些页面
2. 需要更新哪些页面
3. 可能有哪些重复页面
4. 每个页面预计写入什么内容
5. 风险是什么
6. 等我确认后再执行
```

## 14. 让它自动分类

```
请把下面内容自动分类并写入 wiki。

要求：
1. 自动判断 area
2. 自动判断页面类型：source / concept / entity / comparison / overview / insight
3. 如果分类不确定，选择较宽泛 area，并在页面中标记 uncertainty
4. 不要询问我，除非会产生大规模结构变化
5. 写入后更新 index 和 log

内容：

<粘贴内容>
```

## 15. 学习笔记沉淀模板

这个适合你学 Java、Spring AI、LangChain4j。

```
这是我的学习笔记，请整理进 wiki。

学习主题：
<主题>

笔记内容：

<粘贴内容>

要求：
1. 不要原样整理成流水账
2. 提取核心概念
3. 补充定义、使用场景、常见误区
4. 如果涉及技术对比，创建或更新 comparison
5. 如果是一个阶段性主题，更新 overview
6. 保留我的理解，但优化表达
7. 更新 index 和 log
```

## 16. 概念深化模板

当你觉得某个页面太浅，用这个。

```
请深化这个概念页：

wiki/concepts/<area>/概念_xxx.md

要求：
1. 保留已有有效内容
2. 补充清晰定义
3. 补充为什么重要
4. 补充核心机制
5. 补充至少 1 个例子
6. 补充常见误区
7. 补充相关概念链接
8. 补充来源
9. 更新 log
```

---

## 17. 从对话中写回知识库

适合你和 Agent 聊完后，让它沉淀。

```
请把我们刚才这轮对话中有长期价值的内容写回 wiki。

要求：
1. 不要保存闲聊内容
2. 只提取可复用知识
3. 判断应该更新 concept、comparison、overview 还是 insight
4. 优先更新已有页面
5. 不要重复创建页面
6. 更新 index 和 log
7. 输出写回清单
```

---

## 18. 最常用的万能模板

你平时懒得想，就用这个。

```
请把下面内容编译进我的 Obsidian wiki。

内容：

<粘贴内容>

要求：
1. 这不是普通摘要任务，而是知识库编译任务
2. 自动判断模式、领域 area 和页面类型
3. 优先更新已有页面，避免重复创建
4. 至少生成或更新一个有长期价值的 wiki 页面
5. 如果出现重要概念，不要只列名字，要写成可复用 concept note
6. 添加有上下文说明的 wikilink
7. 更新 wiki/index.md
8. 更新 wiki/log.md
9. 最后输出本次修改清单
```

---

我建议你日常主要用这 4 个：

```
1. 请把下面内容编译进我的 Obsidian wiki
2. 请基于 raw/xxx 执行 Ingest
3. 请对主题 xxx 执行 Synthesis
4. 请对当前 wiki 执行 Lint
```
